## 1. 全体方針

### 1-1. 誰に何を約束するか（1段落）

**中小〜中堅の売買仲介の経営者（2〜20名）**に向けて、楽マッチ AI は
「**物件集めから追客・契約まで、新人でもトップ営業の“動き”をなぞれる。しかも、お客様が見つけた物件まで担当者に集まる**」状態を、月3,000円/人（税込）で約束する。
入口の価値は「貼るだけ（コピペ/PDF）で物件登録〜マッチングが終わる手軽さ×低単価」。
記憶に残す堀は「**お客様連動アプリ（お客様→担当者の逆方向）**」と「**逆引きマッチング**」。作文AIは差別化に使わない。

### 1-2. baseline（現状LP）の散らかり診断

現状 `page.tsx` を実読して確認した散らかりの正体（team-lead 一次診断を裏取り）:

- **Hero が文字の壁**: バッジ2個＋H1＋エヤブロウ的サブ1行＋リード＋価格バッジ＋「個人でも会社でも」段落＋CTA2個＋注記＋ベネフィット3カード＋デバイスモック＝1スクロール前に**9ブロック**。Linear/Stripe は above-the-fold が「H1＋サブ＋1〜2CTA＋1ビジュアル」の4要素に絞られている。**現状は要素過多で焦点が定まらない。**
- **価格と CTA の過剰反復**: 「¥3,000」がHero・FeatureHub・利用形態・Pricing・各所で**5回以上**。「1週間無料で試す」系CTAがHero・FeatureHub・Pricing×2・最終CTAで**6箇所**。出す回数と置き場所が無設計。
- **セクションが多く長い（全12）**: Hero / Stats / FeatureHub / 開発者ストーリー / Pain / お客様アプリ / 機能zigzag4本 / 通話録音 / 比較表 / 信頼+AI進化 / 利用形態 / 料金 / FAQ11問 / 最終CTA。**「個人でも会社でも」というメッセージが Hero・利用形態・FAQ・Pricing 注記の4箇所で重複**。zigzag機能4本＋FeatureHub＋お客様アプリ＋通話録音で**機能説明が7セクションに分散**し、何が主役か埋もれる。
- **ベネフィットが機能名に埋もれる**: 「双方向マッチング」「契約フェーズ管理」等、見出しが機能名スタート。Notion は「一括検索ですべてが見つかる」「毎回完璧なメモを」と**状態の変化**を先に出す。
- **社会的証明が弱いまま長い**: 開発者ストーリーはあるが、Notion/Linear のような「他者の言葉・数字」がゼロ。長さの割に信頼の決定打がない。

### 3-0. 日本語の見出しの語り口（headline voice・全セクション共通ルール）

日本のB2B SaaS（NotePM／Stock／Lark／BuildIn AI）の実査から、国内経営者に刺さる見出しの型を抽出した。本ブループリントの全H2はこの型に従っている。

- **成果・状態の変化を主語にする（機能名は後）**: NotePM「社内の知りたい、すぐ見つかる。」、Stock「最も簡単に管理できる」、Lark「仕事をラクにシンプルに」。
  → 楽マッチ採用例: 堀②は「双方向マッチング」（機能名）ではなく「**この物件、誰に当てるか。一瞬で分かる。**」（状態）。AI統合は「専属AI」ではなく「**新人が、初日から動ける。**」（状態）を先に。
- **「成果1行＋機能名1行」の2段見出し**（NotePMの型）: 1行目に状態の変化、2行目に何の機能か。本ブループリントのH2はこの2行構成（例: 堀①「お客様が見つけた物件が、／担当者に集まる。」、AI統合「新人が、初日から動ける。／顧客1人1人に、専属AI。」）。
- **単一ベネフィットを軸にブレさせない**: 楽マッチの一本軸は「**貼るだけ・誰でも回る手軽さ**」。H1・Stats・FeatureHub・開発者の声で同じ“手軽さ／誰でも”を反復する（Stock流）。
- **句点で言い切る短文**: 「〜できます」より「〜する。」「〜が、集まる。」と体言・終止で短く切る方が信頼的で読みやすい（Linear/NotePMの簡潔さ）。本文の説明文（リード）は丁寧体（です・ます）のままでよい＝見出しは断ち、本文は丁寧、のコントラスト。
- **短い動詞を句点で畳みかける型も可（任意）**: BuildIn AI「作成。整理。協働。収益化。」、Microsoft Loop「共に考え、計画し、創造する」のように動詞を句点/読点区切りで連打すると、機能の幅を一息で伝えられる。楽マッチでサブ的に使うなら「貼る。当てる。任せる。」のような体言・動詞連打が候補（H1には使わず、FeatureHub周辺やセクション内の小見出しでの選択肢として）。多用は散らかるので1ページ1回まで。
- **感情的ペイオフを先に、機能は“証拠”として後出し（Arc流）**: Arc は「Enter your new home on the internet」「brought order to the chaos」と**感情・状態の救済**を先に出し、機能（Spaces等）を後から“証拠”として見せる。楽マッチへの翻訳＝H1「入社初日の新人が、トップ営業の“動き”をする。」は既にこの「状態の救済」型。本文では「事務に追われず、商談に集中」「記憶や勘に頼らず、その場で」のような**現場の安堵**を1フレーズ添えてよい（ただし成果保証はNG・§COMPLIANCE）。

### 5-2. タイポ階層（DESIGN_SYSTEM §3 を厳守）

- H1: `text-4xl sm:text-5xl lg:text-6xl font-bold tracking-tight leading-[1.1] text-ink-900 break-keep`（1ページ1回）
- H2: `text-3xl sm:text-4xl font-bold tracking-tight leading-snug text-ink-900 break-keep`

- リード: `text-lg sm:text-xl leading-relaxed text-ink-700`、本文 `text-base leading-relaxed text-ink-700`、注記 `text-sm text-ink-500`
- 本文幅は `max-w-2xl`（日本語38〜42字折返し）。Hero コピーは `max-w-3xl`。`font-bold` は H1/H2/数値のみ。

### 5-3. CTA の出し方（反復の整理）

現状6箇所→**意味のある3箇所＋FeatureHub内1**に整理:
- **Hero**（主CTA `/try` ＋ 副 `/login`）= 最重要。
- **FeatureHub 内**（`無料で試す` /try）= コンポ内既定、触らない。
- **料金**（各プラン `/signup`）= 申込直結。
- **最終CTA**（`/try` ＋ 詳細リンク）= フィナーレ。
- → 堀・AI・信頼セクションの末尾は「詳しく見る」テキストリンク（`/features/*`）のみにし、**ボタンCTAを増やさない**（散らからせない）。
- ボタンは `rounded-xl`、影 `shadow-sm`→hover `shadow-md`、`hover:scale-[1.02]` ＋ `motion-reduce` 対応（現状クラス踏襲）。`rounded-full` は使わない。

### 5-4. モーション（DESIGN_SYSTEM §6）

- `FadeIn` は各セクション初出の登場のみ。1セクション1〜2回。カードは stagger（50〜80ms）。
- ホバーは色/影/微小 translate のみ（150〜200ms）。`scale` は最大 1.02。バウンス・回転禁止。
- `prefers-reduced-motion` 尊重（`motion-reduce:transition-none` / `motion-reduce:hover:scale-100`）。
- FeatureHub の出現アニメ・接続線は現状のまま（触らない）。

### 5-5. notion / AFFiNE / 一流LPから採る具体策（実装の手触り）

- **Notion流モジュール化**: 堀①②・AI・信頼の各セクションは「エヤブロウ→H2→リード→箇条書き/カード→詳細リンク」の同一骨格で繰り返す。**骨格を揃えるとセクションが増えても散らからない**（Notionのカード反復、Linearのセクション統一）。
- **Linear流の余白**: H2 と本体の間は `mt-12 lg:mt-16`、エヤブロウ→H2→リードは `space-y-3〜4`。詰めない。
- **Stripe流の浮かせたスクショ**: P1/P3/P4 は `ring-1 ring-surface-200 shadow-[0_20px_60px_rgba(5,57,43,0.12)]`、背景に淡い `primary-50` ぼかし円を1つ（現状の堀①の手法を他のスクショにも適用）。
- **freee流の課題提起**: Problem（3課題）は §2 で削っていないが、現状の Pain セクションを**信頼統合の前ではなく Hero 直後の Stats と FeatureHub の流れを崩さない位置**＝今回は「FeatureHub で全体像→堀→AI」の後、信頼の手前に置くより、**現状どおり Hero 近くに置く**のが筋が通る。→ 実装判断: Pain（課題3カード）は **Stats と FeatureHub の間** に挟むか、**Hero 直後**に置く（team-lead 確認事項・§6）。どちらも1セクション1メッセージは維持。
- **色数の規律**: 白×緑＋ゴールド1点。ゴールドは「人気」バッジと数値強調のみ（現状維持）。面塗りゴールド禁止。
