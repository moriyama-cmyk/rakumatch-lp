# LP v2（AB案）本番移植 — 実装ノート

PORT_SPEC.md（`C:\dev\rakumatch\_lp_v2_design\PORT_SPEC.md`）を基準に、実装中に判断・逸脱した点を記録する。
ブランチ: `lp-v2-ab`。push・deployはしていない（統括の審査待ち）。

## 重大な発見・修正（バグ）

- **夜セクションが背景に埋もれる不具合を発見・修正**。AB案オリジナルは `.ambient`（fixed・z-index:-2）を
  実`<body>`直下の子として置いており、CSS仕様上「bodyの背景がキャンバス背景へ伝播する」特殊規則
  （CSS Backgrounds §body-background）により、bodyの背景が自動的に`.ambient`より下に回っていた。
  本移植では全体を`.lpv2`という普通のdivでラップしたため、この伝播規則が働かず、`.lpv2`自身の背景
  （`var(--paper)`＝明るいペーパー色）が通常の描画レイヤーで`.ambient`（負のz-index）の上に乗ってしまい、
  夜セクションの文字（`--phase-fg`＝クリーム色）が同系色のペーパー背景に埋もれてほぼ読めなくなっていた。
  `.lpv2`に`isolation:isolate`を追加してこのdiv自身に新しいスタッキングコンテキストを作ることで、
  意図した重なり順（`.lpv2`背景 → `.ambient` → header/コンテンツ）を再現し、修正済み（Playwright実機確認済み）。

## 命名・スコープの変更（PORT_SPEC「`.lpv2`スコープ」要件を満たすための調整）

- `html.reduced` → `.lpv2.lpv2-reduced`（`<html>`ではなくラッパーdivにクラス付与。他ページに影響させないため）
- `.reveal` / `.in-view` → `.lpv2-reveal` / `.lpv2-in-view`（汎用的すぎるクラス名の衝突回避）
- `@keyframes pd-*` → `@keyframes lpv2-pd-*`
- `--mono`: Interを読み込まず、システム等幅フォントスタックに変更。PORT_SPECのフォント指定は
  Shippori Mincho / Zen Kaku Gothic New / Noto Sans JPの3書体のみのため。

## CTA配置の判断（trackCta location命名リストへの対応）

PORT_SPECのlocation命名例（`header / hero_night / turn_morning / loop / compare / pricing_standard /
pricing_premium / story / final_cta / sticky_mobile`）は、AB案オリジナルには存在しないCTAを含んでいた
（AB案の実CTAは header・pricing共通1本・final_cta・の3箇所のみ）。命名リストが主要セクション数と
過不足なく一致すること、および森山さんの「売上最優先モード」方針（現行本番LPも多点CTA設計）から、
**各セクションに1つずつCTAを新設する意図**と判断し、以下を追加した:

- 夜ヒーロー（21:00）に控えめなテキストリンク「→ 登録なしで、今の画面を見てみる」（`hero_night`）
- 夜明け（5:12）に「→ 登録なしで、その朝を先に体験する」（`turn_morning`）
  - 転換点（23:58）には追加しなかった。「説明は、朝に。」という溜めの演出を壊さない判断。
- 核ループ末尾に「→ 実物の画面を触ってみる」（`loop`）
- 比較表末尾に同上（`compare`）
- 開発者ストーリー末尾に同上（`story`）
- 料金: AB案は2枚のカード共通で1本のリンクだったが、`pricing_standard`と`pricing_premium`が
  別名で明記されていたため、**カードごとに個別CTA**に分割（旧・共通リンクは削除）。

## コピー・コンテンツの判断

- 「誰に出すか」→「誰に紹介するか」軸の統一は、森山さんの明示指示3箇所（ヒーローh1・STEP2見出し・
  比較表行）に加え、AB案本文に残っていた以下2箇所も同軸で書き換えた（指示「その他...文脈に合わせて
  紹介軸に書き換え」に基づく）:
  - ヒーローsub「次に誰へ、どの物件を出すべきかまでは」→「次に誰に、どの物件を紹介すべきかまでは」
  - 夜明けsub「『誰に出すか』を探さなくていい朝が」→「『誰に紹介するか』を探さなくていい朝が」
- ヒーローh1の改行: 「この物件、誰に紹介するか」に伸びた分、4行に組み替えた
  （今日もまた、／「この物件、／誰に紹介するか」を、／頭の中だけで探している。）。単純に元の3行構成へ
  長い文を押し込むと1行が長くなりすぎるため、鉤括弧の中で改行してリズムを保った。
- STEP2の描写から「緑の」を削除（「登録された物件には、緑の『マッチ○名』バッジが」→「マッチ○名バッジが」）。
  assets班のmanifest.md記載により、実際のプロダクトUIのバッジ色は紫（ラベンダー）で緑ではないと判明したため、
  事実と異なる形容詞を削除した（同様にFeatureDigestの物件管理カードのalt文からも「緑の」を削除）。
- 導入者の声の「正直な注記」: 現行本番の`Voices.tsx`にはコード内コメントとしての匿名化方針はあるが、
  画面に表示される注記文はなかった（過去に自己言及的な弱気文言は意図的に削除された経緯がMEMORYにある）。
  PORT_SPECは「正直な注記も現行から引き継ぐ」と指示しているため、コードのコメントに書かれている実際の
  運用方針（実在の声・粗い匿名化・捏造なし）を要約した一文を新たに書き起こして掲載した
  （「掲載している声は、実際にご利用いただいている方から伺ったものです。個人が特定されないよう属性は
  粗く匿名化しています。捏造した声や口コミは一切掲載していません。」）。

## 画像アセットの配置判断

- assets班のmanifest.md（`public/lpv2/manifest.md`）到着後に判明: 全景7点はSTEP1/2/3・機能ダイジェスト
  4カードに1:1対応するよう生成されており、拡大切り出し4点は「補助図」として想定されていた
  （manifestの「想定の使い所」列）。これを受けて、**全景画像を各ループステップの主図として維持し
  （AB案オリジナルの構図に忠実）、切り出し画像は`detail-crop`という小さな補助サムネイルとして
  ピンリストの下に併記する**設計にした（PORT_SPECの「全画面縮小で細部が潰れる場面は拡大切り出しに
  差し替え」を、"全置換"ではなく"全景を保ちつつ補強"と解釈）。結果、配布された全14点の画像
  （全景7・切り出し4・iPhone3）をすべて使用している。
- ピン（①②）の座標は、全景画像を使う3ステップともAB案オリジナルのcallout位置（top/left/right の%指定）を
  そのまま採用した（同じ画像を使うため座標もそのまま転用できる）。
- iPhone画像の実寸: manifestでは750×1630（AB案オリジナルの868×1887とは異なる）。width/height属性と
  `.phone-frame img`のaspect-ratioを実寸に合わせて修正（縦横比はほぼ同一のため見た目への影響はない）。

## その他の構造上の判断

- 開発者ストーリー節のid: AB案では夜セクションのラッパーdivが`id="story"`を使っているため、
  新設した開発者ストーリー節は`id="founder"`とし、DOM内のid重複を避けた。
- フッター: AB案の3リンク構成（利用規約・プライバシーポリシー・特定商取引法に基づく表記）をそのまま維持し、
  リンク先のみ実在ページ（/terms /privacy /tokusho）に接続。旧`Footer.tsx`にある「お問い合わせ」リンクは
  AB案に存在しないため追加しなかった。
- 比較表のモバイル表示は、JSでの出し分け（ハイドレーション不一致のリスク）を避けるため、CSSの
  `@media`のみで表⇄カードを切り替える設計にした（`table-scroll`は768px未満で`display:none`、
  `compare-cards`は768px以上で`display:none`）。
- 既存の共有ユーティリティ（`src/components/lp/lib/cn.ts`・`src/components/lp/lib/headline.tsx`の`hl()`）を
  そのまま再利用した（新規実装せず）。どちらも特定セクションに紐づかない汎用ヘルパーのため、
  「旧sectionsは削除しない」の対象外と判断。

## 品質ゲート（自己QA）

`npm run build` 成功・`npm run start`をポート4173で起動し、Playwright（scratchpad常設プロジェクト）で
375 / 768 / 1280 / 1536px（reduced-motionは375・1280で追加検証）を実機確認。

- 横スクロール: 全幅・全モードでゼロ（`document.documentElement.scrollWidth === clientWidth`）
- コンソールエラー: ゼロ（サンドボックス環境固有の`net::ERR_CONNECTION_TIMED_OUT`のみ検出。これは
  GA4/Clarityが`googletagmanager.com`/`clarity.ms`への外部通信を試みるもので、この検証環境が外部
  ネットワークに出られないために発生する既知の環境要因。本番Vercel環境では発生しない）
- 比較表: モバイルはカード表示・横スクロールなしで可読。デスクトップは通常の表＋楽マッチ列を緑ハイライト
- 核ループ: デスクトップでスティッキーフレーム＋IntersectionObserverによるステップ切替・ピン切替・
  URL文字列切替・pip切替が正常動作。モバイル／reduced-motionは3ステップ静止画（fallback figure）に
  正しくフォールバック
- STEP1変換実演（ParseDemo）: reduced-motionでは完成形を即表示。通常モードはスクロールイン後に
  アニメーション開始
- 夜→朝の背景色遷移: スクロールに連動して自然に変化することを確認（上記バグ修正後）

## 残課題・引き継ぎ事項

- ピン座標（top/left/right の%指定）は全景スクリーンショットに対する目視合わせであり、実際の
  ブラウザでもう一段階の微調整確認を推奨（現時点でPlaywrightスクリーンショット上ではおおむね
  該当パネルを指せていることを確認済み）。
- モバイル下部固定CTAが最終CTAセクションを通過してフッターに入った後に再表示されるかどうかは、
  既存`StickyCta.tsx`と同一ロジックをそのまま踏襲しており、個別の再検証はしていない
  （既存コンポーネントの実績を信頼した）。
- STEP2（物件詳細）のスティッキーフレーム画像が、スクリーンショット上でやや彩度が低く見える瞬間があった。
  クロスフェード遷移中の一時的なキャプチャの可能性が高いが、実機での目視も推奨。
